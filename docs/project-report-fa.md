# De Gouden Lepel - دستیار عملیات رستوران

## گزارش پروژه — هکاتون Prosus/Toqan

## ۱. هدف پروژه

### مشکل

مدیران رستوران هر روز با سه حوزه عملیاتی حیاتی سر و کار دارند: **مدیریت موجودی**، **سفارشات/درآمد** و **برنامه‌ریزی نیروی کار**. این حوزه‌ها معمولاً در سیستم‌های جداگانه مدیریت می‌شوند و اتصال بینش‌ها بین آن‌ها دشوار است. به عنوان مثال، ممکن است مدیر متوجه نشود که پرفروش‌ترین غذایش به ماده‌ای وابسته است که در آستانه اتمام قرار دارد، یا اینکه در یک روز کم‌مشتری، نیروی کار اضافی دارد.

### راه‌حل

ما یک **دستیار عملیاتی مبتنی بر هوش مصنوعی** برای رستوران De Gouden Lepel (قاشق طلایی) در آمستردام ساختیم. این عامل (Agent) یک رابط مکالمه‌ای واحد ارائه می‌دهد که هر سه حوزه را در قالب اطلاعات یکپارچه و قابل اقدام ترکیب می‌کند.

مدیر رستوران می‌تواند بپرسد:

> "گزارش صبحگاهی من را بده"

و عامل با یک مرور عملیاتی کامل پاسخ می‌دهد: درآمد امروز، پرفروش‌ترین‌های هفته، برنامه کاری کارکنان، غیبت‌ها، هزینه‌های نیروی کار و هشدارهای موجودی — همه به هم متصل. اگر یک آیتم پرفروش به ماده‌ای با موجودی کم وابسته باشد، عامل فوراً آن را اعلام کرده و سفارش مجدد مشخصی پیشنهاد می‌دهد.

### اهمیت برای Toqan

Toqan پلتفرمی برای استقرار عامل‌های هوش مصنوعی در رستوران‌ها با بیش از ۱۰,۰۰۰ کاربر روزانه است. این عامل نشان می‌دهد که هوش بین‌حوزه‌ای چگونه می‌تواند مشکلات عملیاتی واقعی رستوران‌ها را حل کند — نه فقط پاسخ به سؤالات ساده، بلکه استدلال در چندین منبع داده و پیشنهاد اقدامات مشخص.

## ۲. رویکرد پیاده‌سازی

### پشته فناوری

| مؤلفه | فناوری |
|---|---|
| زبان برنامه‌نویسی | Go 1.25 |
| مدل هوش مصنوعی | Claude (از طریق Anthropic Go SDK) |
| فریمورک HTTP | chi v5 |
| رابط ترمینال | BubbleTea (Charm) |
| API تحلیلی | Toqan Cube.js API |
| کانتینرسازی | Docker (ساخت چند مرحله‌ای Alpine) |

### تصمیمات طراحی

**چرا Go؟** زبان Go یک باینری کامپایل‌شده واحد بدون وابستگی‌های زمان اجرا، زمان راه‌اندازی سریع و همزمانی داخلی ارائه می‌دهد. این ویژگی‌ها برای سروری که فراخوانی‌های موازی API انجام می‌دهد و باید برای پلتفرم Toqan کانتینرسازی شود، ایده‌آل است.

**چرا حلقه عاملی (و نه یک چت‌بات ساده)؟** پرس‌وجوهای رستوران ذاتاً چند مرحله‌ای هستند. یک "گزارش صبحگاهی" نیاز به واکشی داده‌های سفارش، پرفروش‌ها، برنامه کارکنان، هزینه‌های نیروی کار، غیبت‌ها و هشدارهای موجودی دارد — و سپس ترکیب آن‌ها. حلقه عاملی به Claude اجازه می‌دهد تصمیم بگیرد کدام ابزارها را به چه ترتیبی فراخوانی کند و پرس‌وجوهای پیچیده را بدون کدنویسی سخت منطق مدیریت کند.

**چرا فراخوانی ابزار به جای RAG؟** داده‌ها ساختاریافته و عملیاتی هستند (اعداد، برنامه‌ها، سطح موجودی). فراخوانی ابزار به Claude داده‌های دقیق و به‌روز از APIها و انبار موجودی می‌دهد، به جای جستجو در اسناد.

**چرا موجودی درون‌حافظه‌ای؟** برای هکاتون، وضعیت درون‌حافظه‌ای با داده‌های اولیه به ما امکان می‌دهد کل جریان کار موجودی (بررسی موجودی، ثبت تحویل، ثبت ضایعات، ثبت سفارش) را بدون نیاز به پایگاه داده نمایش دهیم. راهنمای پلتفرم Toqan شامل جایگزین Google Sheets برای ماندگاری است.

### حلقه عاملی

این هسته اصلی سیستم است. وقتی کاربر پیامی ارسال می‌کند:

```
۱. پیام کاربر به تاریخچه جلسه اضافه می‌شود
۲. حلقه (حداکثر MaxAgentIterations = ۱۵):
   الف. ارسال کل تاریخچه مکالمه + تعاریف ابزار به Claude
   ب. Claude با یکی از دو حالت پاسخ می‌دهد:
      - فقط متن    --> بازگشت پاسخ به کاربر (پایان)
      - فراخوانی ابزار --> اجرای هر ابزار از طریق Registry
   پ. نتایج ابزار به عنوان پیام کاربر به تاریخچه اضافه می‌شود
   ت. بازگشت به مرحله (الف)
۳. اگر به حداکثر تکرار رسید، خطا برگردانده می‌شود
```

این به Claude اجازه می‌دهد چندین فراخوانی ابزار را زنجیره کند. به عنوان مثال، یک "گزارش صبحگاهی" ابزار `get_daily_briefing` را فعال می‌کند (که ۵ فراخوانی موازی API انجام می‌دهد)، و Claude ممکن است سپس با `list_low_stock` برای بررسی آیتم‌های مرتبط ادامه دهد.

### مدل دوگانه داده

عامل با دو منبع داده اساساً متفاوت کار می‌کند:

| جنبه | موجودی (محلی) | عملیات (راه دور) |
|---|---|---|
| ذخیره‌سازی | نقشه‌های درون‌حافظه Go | Toqan Cube.js API |
| قابلیت تغییر | خواندن + نوشتن | فقط خواندن |
| ماندگاری | با راه‌اندازی مجدد ریست می‌شود | دائمی (سرویس خارجی) |
| انواع داده | سطح موجودی، آیتم‌های منو | سفارشات، درآمد، شیفت‌ها، غیبت‌ها |
| تعداد ابزار | ۸ ابزار | ۵ ابزار |
| همزمانی | محافظت با Mutex | فراخوانی HTTP (تایم‌اوت ۱۵ ثانیه) |

### مسئولیت بسته‌ها

**`cmd/agent/main.go`** — نقطه ورود سرور HTTP. تمام مؤلفه‌ها را به ترتیب ایجاد و به هم متصل می‌کند: Config -> MemoryStore -> ToqanClient -> ToolRegistry -> LLMClient -> Agent -> HTTPRouter. خاموشی ملایم از طریق سیگنال‌های سیستم‌عامل را مدیریت می‌کند.

**`cmd/tui/`** — رابط ترمینال تعاملی ساخته شده با BubbleTea. یک رابط چت با حباب‌های پیام استایل‌دار، اسپینر در حین تفکر عامل و رندر Markdown پاسخ‌های عامل از طریق Glamour ارائه می‌دهد.

**`pkg/agent/agent.go`** — شامل ساختار Agent و متد اصلی `Chat()` است که حلقه عاملی را پیاده‌سازی می‌کند. تاریخچه مکالمه هر جلسه را در `map[string][]MessageParam` با قفل mutex اختصاصی هر جلسه برای ایمنی همزمانی مدیریت می‌کند. جلسات پس از ۱ ساعت بیکاری با goroutine پاکسازی پس‌زمینه منقضی می‌شوند.

**`pkg/llm/client.go`** — پوشه SDK Anthropic Go. سه بخش حیاتی:

1. `GenerateSchema[T]()` — از بازتاب `invopop/jsonschema` برای تولید schema‌های JSON از ساختارهای Go برای تعاریف ورودی ابزار استفاده می‌کند.
2. `BuildToolDefinitions()` — تمام ۱۳ ابزار را با نام، توضیحات و schemaها تعریف می‌کند. این یک نیمه از "قرارداد" ابزار است.
3. `SystemPrompt` — پرامپت سیستم کامل که به Claude نقش، قابلیت‌ها و راهنماهایش را آموزش می‌دهد.

**`pkg/tools/registry.go`** — یک `map[string]ToolFunc` ساده که نام ابزار را به توابع مدیریت نگاشت می‌کند. متد `Execute()` هندلر مناسب را پیدا و فراخوانی می‌کند. این نیمه دیگر "قرارداد" ابزار است.

**`pkg/tools/inventory.go`** — ۸ ابزار موجودی را ثبت می‌کند. هر هندلر ابزار ورودی JSON را به یک ساختار Go تایپ‌شده تبدیل، متد مناسب MemoryStore را فراخوانی و یک رشته JSON نتیجه برمی‌گرداند.

**`pkg/tools/operations.go`** — ۵ ابزار عملیاتی را ثبت می‌کند که API خارجی Toqan را پرس‌وجو می‌کنند. ابزار `get_daily_briefing` قابل توجه است: ۵ goroutine برای فراخوانی‌های موازی Cube.js راه‌اندازی می‌کند (سفارشات، آیتم‌های برتر، شیفت‌ها، هزینه‌های نیروی کار، غیبت‌ها)، نتایج را از طریق channel جمع‌آوری و هشدارهای موجودی کم محلی را اضافه می‌کند.

**`pkg/toqan/client.go`** — کلاینت HTTP برای API تحلیلی Toqan Cube.js. درخواست‌های `POST /cubejs-api/v1/load` با پرس‌وجوهای ساختاریافته (اندازه‌گیری‌ها، ابعاد، بازه‌های زمانی، فیلترها) ارسال می‌کند. شامل `PeriodToDateRange()` است که دوره‌های انسانی ("today"، "this_week") را به آرایه‌های بازه تاریخ تبدیل می‌کند.

**`pkg/api/`** — مسیریابی HTTP از طریق chi v5 با میان‌افزار (شناسه درخواست، لاگ‌گیری، بازیابی، تایم‌اوت ۱۲۰ ثانیه، CORS). هندلرها سبک هستند: درخواست را تجزیه، عامل یا انبار را فراخوانی و JSON برمی‌گردانند.

**`pkg/config/config.go`** — پیکربندی را از متغیرهای محیطی با مقادیر پیش‌فرض منطقی بارگذاری می‌کند. از `godotenv` برای بارگذاری خودکار فایل‌های `.env` استفاده می‌کند.

**`internal/store/memory.go`** — MemoryStore سه ساختار داده را پشت یک `sync.RWMutex` نگه می‌دارد: آیتم‌های موجودی (۲۵ ماده)، آیتم‌های منو (۸ غذا با دستور مواد) و سفارشات (سفارشات مجدد ثبت شده). تمام کلیدهای نقشه با حروف کوچک هستند. متد `seed()` با داده‌های واقعی De Gouden Lepel پر می‌شود.

## ۵. مرجع API

### POST /api/v1/chat

ارسال پیام به عامل و دریافت پاسخ.

**درخواست:**
```json
{
  "message": "Give me my morning briefing",
  "session_id": "optional-session-id"
}
```

**پاسخ:**
```json
{
  "response": "Here is your morning briefing for today...",
  "session_id": "auto-generated-uuid-if-not-provided",
  "tool_calls": 3
}
```

`session_id` زمینه مکالمه را بین درخواست‌ها حفظ می‌کند. اگر حذف شود، یک UUID جدید تولید می‌شود. فیلد `tool_calls` نشان می‌دهد عامل چند ابزار فراخوانی کرده است.

### GET /api/v1/inventory

تمام آیتم‌های موجودی را با تعداد برمی‌گرداند.

**پاسخ:**
```json
{
  "items": [
    {
      "name": "biefstuk",
      "quantity": 6,
      "unit": "kg",
      "reorder_threshold": 8,
      "category": "meat",
      "supplier": "Slagerij de Wit",
      "last_updated": "2026-03-02T10:00:00Z"
    }
  ],
  "count": 25
}
```

### DELETE /api/v1/sessions/{sessionID}

تاریخچه مکالمه یک جلسه چت را پاک می‌کند.

### GET /health

`{"status": "ok"}` برمی‌گرداند.

| `get_daily_briefing` | (بدون ورودی) | گزارش جامع: ۵ فراخوانی موازی API + هشدارهای موجودی |

**مقادیر دوره:** `today`, `yesterday`, `this_week`, `last_7_days`, `this_month`, `last_30_days`

## ۷. سرویس‌ها و یکپارچه‌سازی‌های خارجی

### API Claude (Anthropic)

عامل از Claude به عنوان موتور استدلال خود از طریق Anthropic Go SDK (`anthropics/anthropic-sdk-go` نسخه ۱.۲۶.۰) استفاده می‌کند. هر درخواست چت، کل تاریخچه مکالمه به علاوه ۱۳ تعریف ابزار را ارسال می‌کند. Claude بر اساس سؤال کاربر تصمیم می‌گیرد کدام ابزارها را فراخوانی کند.

**الگوی استفاده از SDK:**
```go
resp, err := client.API.Messages.New(ctx, anthropic.MessageNewParams{
    Model:     anthropic.Model("claude-sonnet-4-5-20250929"),
    MaxTokens: 4096,
    System:    []anthropic.TextBlockParam{{Text: llm.SystemPrompt}},
    Messages:  history,
    Tools:     toolDefinitions,
})
```

### API Toqan Cube.js

API تحلیلی Toqan داده‌های عملیاتی رستوران را در قالب Cube.js ارائه می‌دهد. تمام پرس‌وجوها به `POST /cubejs-api/v1/load` با هدر کلید API ارسال می‌شوند.

**مکعب‌های Cube.js موجود:**
- `Orders` — تعداد سفارش، درآمد، انعام، تخفیف، لغوها
- `Revenue` — درآمد هر آیتم و تعداد فروش
- `Shifts` — برنامه کارکنان، ساعات، بخش‌ها
- `Costs` — هزینه‌های نیروی کار بر اساس بخش
- `Absences` — غیبت‌های کارکنان با وضعیت تأیید

**نمونه پرس‌وجو:**
```json
{
  "query": {
    "measures": ["Orders.count", "Orders.totalRevenue"],
    "timeDimensions": [{
      "dimension": "Orders.orderedAt",
      "dateRange": ["2026-03-02", "2026-03-02"]
    }]
  }
}
```

## ۸. همزمانی و طراحی جلسه

### مدیریت جلسه

هر جلسه چت با یک UUID شناسایی می‌شود و تاریخچه مکالمه خود را حفظ می‌کند. ویژگی‌های طراحی کلیدی:

- **قفل اختصاصی جلسه:** یک `sync.Mutex` برای هر جلسه تضمین می‌کند که درخواست‌های همزمان به یک جلسه سریالی شوند (جلوگیری از درهم‌آمیختن فراخوانی‌های ابزار).
- **قفل نقشه سراسری:** یک `sync.RWMutex` از نقشه جلسات برای ایجاد و جستجوی ایمن همزمان محافظت می‌کند.
- **انقضای TTL:** جلسات بیکار بیش از ۱ ساعت به طور خودکار توسط goroutine پس‌زمینه‌ای که هر ۱۰ دقیقه اجرا می‌شود، پاکسازی می‌شوند.

### فراخوانی‌های موازی API

ابزار `get_daily_briefing` مدل همزمانی Go را نمایش می‌دهد: ۵ goroutine (یکی برای هر پرس‌وجوی Cube.js) راه‌اندازی می‌کند، نتایج را از طریق channel بافرشده جمع‌آوری و با استفاده از `sync.WaitGroup` منتظر تکمیل همه می‌ماند. این تأخیر را از ۵ فراخوانی متوالی API به تقریباً زمان کندترین فراخوانی واحد کاهش می‌دهد.

## ۹. استقرار

### Docker

فایل Dockerfile از ساخت چند مرحله‌ای استفاده می‌کند:

1. **مرحله ساخت:** Go 1.25 Alpine یک باینری استاتیک با `CGO_ENABLED=0` کامپایل می‌کند

docker build -t restaurant-agent .
docker run -p 8080:8080 --env-file .env restaurant-agent
